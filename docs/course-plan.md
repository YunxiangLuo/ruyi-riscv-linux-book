# 课程计划表

本计划表由 `docs/course-outline.md` 中已经定好的课程大纲出发，并参考 RuyiSDK 资料、教材评审意见、开发板课程案例和嵌入式 Linux 应用开发资料整理而来。

## 使用原则

- 已定稿的课程大纲不在本计划表中改动；本计划表只负责把大纲拆成可执行的开发计划。
- ch01-ch07 的实验必须成为 ch08 综合项目的基础步骤，不能设计成彼此孤立的一次性实验。
- ch08 汇总前七章能力，完成项目接口对齐、HTTP 状态页和综合验收。
- ch09-ch11 每章都必须形成一个完整扩展项目，不能只停留在知识点介绍。
- ch10 默认项目调整为 TinyML KWS 语音关键词识别控灯，关键词为“开/关/闪”，LED 作为验收输出。
- Face Detection / YOLO 不再作为 ch10 默认最终作品，只作为后续备选或拓展。
- 大模型 / Agent / Workflow 等新增内容先进入扩展项目池，不直接改动已定主线大纲。
- 授课章节应提交：课程文档、授课 PPT。
- 实验章节应提交：实验指导书。
- 三级小节按课时级组织，每章通常 3-4 个 class；细碎 Linux 命令、调试动作和工具选项放入前期实验指导书和课堂步骤，不拆成独立小节。
- Linux 命令基础在 ch01/ch02 实验指导书中递进体现；ch02 增加 Git 基础；调试方法优先放入实验指导书；systemd 部署基础在 ch01 以服务状态观察的方式点到为止。
- 完成后在对应交付物列标记状态。
- 建议状态：`未开始`、`进行中`、`已完成`、`待审核`、`需修改`、`终审通过`。
- `计划完成时间`、`实际提交时间`、`审核日期`、`责任人` 可根据项目管理安排补充。

## 维护规则

- 课程大纲是基准，不因计划表、资料或临时想法随意改变。

- 修改章节目录后，应同步更新“课程开发进度总表”的 `章节目录` 列。
- 维护知识点到实验映射时，ch01-ch07 每个实验都必须标注对应的 ch08 复用点。

- 每章完成课程文档、PPT 或实验指导书后，应及时更新对应状态。
- 审核意见应具体到章节、三级小节、PPT 页码、实验步骤、运行验证或代码位置。
