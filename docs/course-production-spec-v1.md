# 课程制作规范 V1.0

本规范用于统一 Ruyi RISC-V Linux 课程的多人协作生产标准。后续课程文档、PPT、实验指导书、代码工程和试讲材料均以本规范为准。

## 目标

- 让多人分工写作时保持结构、深度、风格和交付物一致。
- 先定义标准，再批量生产课程内容。
- 保证课程“入门门槛低，出门能力高”。
- 以 K1 / Muse Pi Pro 为课程目标板，当前 K1 暂不可用时可用 LicheePi 4A 做临时验证，QEMU 只承担低硬件依赖的前置实验。
- 先完成一个完整样章，再根据反馈扩展全书内容。

## 一节课标准结构

每节课对应一个 `class*.md` 和一套 PPT。每节课建议包含 3 个左右知识点。

| 项目 | 标准 |
| --- | --- |
| 知识点数量 | 2-4 个，建议 3 个 |
| 课程文档 | 1 份 `class*.md` |
| PPT | 25-30 页左右 |
| 实验 | 1-3 个，单个实验建议 30 分钟以内 |
| 代码 | 只放必要示例和实验代码，避免大段无解释代码 |
| 图表 | 每个重点知识点至少 1 张图、流程图、结构图或运行截图 |
| 练习 | 每节课至少 1 个课堂练习或思考题 |
| 运行验证 | 必须有可观察结果 |

## 课程文档规范

每个 `class*.md` 必须包含：

1. 本章目标
2. 前置条件
3. 知识简介
4. 环境准备
5. 操作步骤
6. 运行验证
7. 常见问题
8. 本章成果

### 内容要求

| 项目 | 标准 |
| --- | --- |
| 目标 | 3 条左右，必须可验证 |
| 知识点 | 2-4 个，建议 3 个 |
| 原理说明 | 每个重点知识点必须解释“为什么”和“怎么用” |
| 图表 | 每个重点知识点建议配图，图必须有编号和解释 |
| 代码 | 代码后必须有逐段说明和运行验证 |
| 练习 | 每节课至少 1 个练习或思考题 |
| 常见问题 | 至少 2 个，来自真实调试或可预见错误 |
| 成果 | 必须能对应 `docs/course-plan.md` 的运行验证 |

### 写作风格

- 面向本科或初学者，先讲使用场景，再讲操作步骤。
- 技术名词保留英文，第一次出现时解释含义。
- 命令、路径、函数名、文件名使用 Markdown 行内代码。
- 不整段搬运第三方资料；引用资料时标明来源。
- 不把扩展方向内容提前塞入主线章节。

## PPT 规范

每节课 PPT 建议 25-30 页，和课程文档一一对应。

| 页面类型 | 建议页数 | 内容 |
| --- | --- | --- |
| 封面与目标 | 2-3 页 | 标题、本节目标、学习成果 |
| 知识点 1 | 6-8 页 | 问题引入、原理图、案例、操作或代码 |
| 知识点 2 | 6-8 页 | 问题引入、原理图、案例、操作或代码 |
| 知识点 3 | 6-8 页 | 问题引入、原理图、案例、操作或代码 |
| 实验说明 | 2-4 页 | 实验目标、步骤、运行验证、提交要求 |
| 小结 | 1-2 页 | 本节成果、常见错误、下一节连接 |

### PPT 要求

- PPT 内容必须来自课程文档，不出现讲义中没有解释的概念。
- 每个重点、难点理论知识应配图表。
- 每个实践操作应配步骤、截图或代码片段。
- 单页避免大段文字。
- 图片必须框出关键区域并编号。
- 代码页只展示核心片段，完整代码放仓库。

## 实验指导书规范

每节课建议 1-3 个实验，单个实验控制在 30 分钟以内。

每个实验必须包含：

1. 实验标题
2. 实验目标
3. 知识点讲解
4. 实验环境
5. 示例
6. 实验任务
7. 实验步骤
8. 运行验证
9. 常见问题
10. 提交要求

### 实验分类

| 类型 | 用途 | 要求 |
| --- | --- | --- |
| 演示实验 | 教师课堂演示 | 结果直观、步骤短 |
| 必做实验 | 学生必须完成 | 能独立复现，有明确验收 |
| 拓展实验 | 学有余力或课程设计 | 标注拓展，不阻塞主线 |

## 代码工程规范

每个需要代码的实验应尽量遵循统一结构：

```text
project/
├── README.md
├── Makefile
├── src/
├── include/
├── config/
├── data/
└── logs/
```

最低要求：

- 有 `README.md` 说明构建、运行和验证。
- 有统一构建入口，例如 `make build`。
- 有清理入口，例如 `make clean`。
- 代码注释说明关键逻辑，不写无意义注释。
- 运行结果必须能对应实验指导书的运行验证。
- 自动化脚本如需保留，优先放在项目外部的内部工程目录；公开仓库中的代码工程不强制包含 `scripts/`。

## 开发板与环境规范

- 课程目标板：K1 / Muse Pi Pro。
- 临时验证板：LicheePi 4A，仅用于 K1 到位前的样章和早期实验验证。
- QEMU 用于 ch01、ch02 以及文件、线程等低硬件依赖内容的前置验证。
- 真实外设、传感器、摄像头、AI 和板级配置最终必须回到 K1 / Muse Pi Pro。
- 不同时展开 Duo、Duo256、K230、BPI-F3 等多板路线；LicheePi 4A 只作为阶段性验证环境，不进入课程主线板卡。
- 如果后续支持多板，只能作为附录或拓展适配，不影响主线。
- 样章或实验使用 LicheePi 4A 验证时，必须在文档中标注实际验证环境，并把板级差异记录到 `boards/licheepi4a/`。

## 成长曲线要求

课程必须体现能力递进：

| 阶段 | 学生能力 |
| --- | --- |
| 入门 | 能进入系统、编译程序、控制 GPIO |
| 基础 | 能使用 UART、PWM、ADC、OLED、传感器 |
| 应用 | 能写文件、配置、日志、多线程程序 |
| 项目 | 能完成联网信息采集终端 |
| 扩展 | 能接入 Camera、TinyML 边缘推理、大模型或系统扩展方向 |

不允许出现“第一章会点灯，最后一章还是只会点灯”的课程设计。

## 样章生产要求

在批量编写前，先完成：

- 1 个完整样章
- 1 套 PPT
- 1 套实验指导书
- 1 套可运行代码

样章完成后，根据实际产出修正计划和模板。
